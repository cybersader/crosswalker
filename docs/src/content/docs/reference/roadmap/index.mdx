Crosswalker is a **meta-system for [ontology lifecycle](/crosswalker/concepts/ontology-lifecycle/) management**, not just a framework importer. The roadmap reflects this — architecture decisions come first, features are built on that foundation, and every significant decision is [logged with rationale](/crosswalker/agent-context/zz-log/).

See [vision alignment decisions](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/) for the philosophical foundations.

:::note[How this roadmap is maintained]
This page is the **living roadmap** — only active and future phases live here. When a phase completes:

1. Move its checklist to a new archive page: `reference/roadmap/vX-Y-name.mdx`
2. Add a "What carried forward" section noting items that moved to later phases
3. Link the archive from the [Archive](#archive) section below
4. Update `ROADMAP.md` at the repo root to stay in sync

Every significant decision gets a dated entry in the [decision log](/crosswalker/agent-context/zz-log/). Link roadmap items to their log entries so the *why* is always reachable from the *what*.
:::

## Current phase: Foundation

## Foundation — "Get the architecture right"

Architecture decisions before more code. The three interconnected schemas (_crosswalker metadata, [FrameworkConfig](/crosswalker/agent-context/config-schema-design/), EvolutionPattern) must be designed together — getting any wrong is expensive to fix later. [Decision: architecture first](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#4-architecture-decisions-before-more-ui-code)

**Research & spec** (iterative, philosophical, first-principled):

- [ ] **[Ontology diff primitives](/crosswalker/agent-context/zz-log/2026-04-09-atomic-operations-research/)** — define the [9 atomic operations + 4 recognized composites](/crosswalker/agent-context/zz-log/2026-04-09-atomic-operations-research/) (also known as: graph change atoms, structural change primitives). [Provably complete](/crosswalker/agent-context/zz-log/2026-04-09-atomic-operations-research/) per graph edit distance literature. The mathematical foundation — "what changed?" [First principles](/crosswalker/agent-context/zz-log/2026-04-08-ontology-evolution-first-principles/) · [Glossary](/crosswalker/concepts/zz-glossary/)
- [ ] **[EvolutionPattern](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) vs transformation recipes** — investigate whether the [EvolutionPattern taxonomy](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) (also: framework handling profile, stewardship profile) is needed alongside per-version-transition transformation records. Could storing the actual transformation per version change (e.g., "NIST r5→r6: these IDs renamed, these merged") replace or complement the framework-level classification? Both may coexist — [research item](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/). [User-first maintenance](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/)
- [ ] **[SEACOW](https://github.com/cybersader/cyberbase) + [folder-tag-sync](https://github.com/cybersader/obsidian-folder-tag-sync) prior-art integration** — the project owner has already built two tools that wrestle with the hierarchy-vs-graph tension Crosswalker is facing. **SEACOW** is a meta-framework for knowledge organization inside filesystem primitives (lives in `cyberbase`), and **folder-tag-sync** is a working Obsidian plugin that layers polyhierarchy onto the filesystem via synchronized tags. Decide what Crosswalker reuses (SEACOW vocabulary, folder-tag-sync regex engine, the "primary folder + parallel tag hierarchy" UX pattern) vs. lets evolve in parallel. Research item — see [04-10 SEACOW + folder-tag-sync prior-art log](/crosswalker/agent-context/zz-log/2026-04-10-seacow-and-folder-tag-sync-prior-art/).
- [ ] **Evidence-framework edge model (the *other* edge type)** — **Direction committed** per the [2026-04-10 evidence-link edge model synthesis log](/crosswalker/agent-context/zz-log/2026-04-10-evidence-link-edge-model-synthesis/), which distilled 2 research sessions targeted at [Challenge 07](/crosswalker/agent-context/zz-challenges/07-link-metadata-edge-model/). Evidence links are stored as **junction notes** (one markdown file per edge, edge-as-note reification) with a 13-field flat-YAML frontmatter schema isomorphic to [OSCAL](/crosswalker/reference/registry/oscal/)'s `by-component` assembly. Three orthogonal status dimensions: `status` (5 OSCAL values: implemented/partial/planned/alternative/not-applicable) + `freshness` (4 computed values: fresh/stale/expired/not-set) + `evidence_type` (8 values: policy-document, automated-scan, test-result, configuration, log-export, interview, architecture-review, attestation). Git history with 3 hardening measures (signed commits, branch protection, periodic external snapshots) is the audit trail. [SSSOM](/crosswalker/reference/registry/sssom/) dual-confidence scoring is borrowed. The original [link metadata system](/crosswalker/agent-context/link-metadata-system/) inline-Dataview syntax is **superseded** by this commitment. **Remaining open sub-decisions** (design work, not research): (1) UUID enterprise resilience scope, (2) Tier 2 sql.js sidecar SQL schema mapping, (3) [synthetic spine (Challenge 06)](/crosswalker/agent-context/zz-challenges/06-synthetic-spine/) integration question — direct or through-spine, (4) tested vault scale threshold documentation, (5) mandatory vs optional field marking, (6) multi-editor conflict resolution, (7) inline-Dataview migration script for v0 vaults.
- [ ] **Evidence mapping workflow** — the broader surface around the edge model above: how evidence gets imported / authored / linked, how status transitions work (covered → partial → planned → deprecated), how reviewer attribution and review dates are captured, how coverage gaps are surfaced, and what the UX looks like for a GRC analyst mapping a policy document to multiple controls across multiple frameworks. The [Python tool's original link-metadata system](/crosswalker/agent-context/link-metadata-system/) sketched one approach; the Obsidian plugin needs a refreshed version informed by the edge-model decision above and the [Obsidian Bases direction research](/crosswalker/reference/roadmap/) (how evidence queries get surfaced as tables/dashboards). [Evidence mapping for GRC teams](/crosswalker/getting-started/grc-teams/) is the user-facing slice; this roadmap item tracks the design commitment behind it.
- [ ] **Crosswalk edge semantics commitment (STRM + SSSOM)** — lock in the vocabulary and metadata envelope every crosswalk edge must carry. The [04-10 synthesis](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#edge-semantics-should-be-set-theoretic-not-skos-ish) identified two layers that work together: **[STRM](/crosswalker/reference/registry/strm/)** (the 5 set-theory relationships from NIST IR 8477 + SCF + OSCAL) as the **required `predicate_id` vocabulary**, and **[SSSOM](/crosswalker/reference/registry/sssom/)** (Matentzoglu et al., *Database*, 2022) as the **row-schema envelope** that every edge sits inside — mandatory `mapping_justification`, optional `confidence`, `author_id`, `mapping_date`, `mapping_tool`, and `predicate_modifier` for negation. [SKOS](/crosswalker/reference/registry/skos/) is rejected as the base vocabulary (deliberately too vague for compliance-grade audit) but kept as an export format via [YAML-LD / LinkML](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#file-first-is-valid-at-current-scale). Decision has to be made before any crosswalk authoring features ship.
- [ ] **Pairwise crosswalks vs synthetic spine architecture** — the biggest open architectural question from the [04-10 foundation synthesis](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#the-synthetic-spine-is-the-missing-architectural-insight). Every mature compliance meta-framework (SCF, UCF, DESM, Hyperproof) converged independently on hub-and-spoke mapping through a canonical intermediate ontology (the "spine") instead of O(n²) pairwise crosswalks. Investigate whether Crosswalker should adopt a spine, which candidate (inherit SCF, distill via FCA, handcraft), and critically — **the long-term resilience and audit-grade trustworthiness** of synthetic mapping spines (governance longevity, error-propagation blast radius, provenance-chain trust for transitively-derived crosswalks, exit strategy if the spine maintainer disappears). [Research challenge brief](/crosswalker/agent-context/zz-challenges/06-synthetic-spine/) · [Synthesis log section](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#the-synthetic-spine-is-the-missing-architectural-insight)
- [ ] **[FrameworkConfig v2 schema](/crosswalker/agent-context/config-schema-design/)** — evolution metadata, sheet selection, custom transforms, crosswalk definitions. Config-as-code format (JSON/YAML) as [source of truth](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#2-config-as-code-is-the-source-of-truth); UI edits config files. [Current schema design](/crosswalker/agent-context/config-schema-design/)
- [ ] **[_crosswalker metadata v2](/crosswalker/agent-context/constraint-enforcement/#metadata-tiers)** — version tracking, source hash, status field, evolution profile reference, migration history. Progressive [metadata tiers](/crosswalker/agent-context/constraint-enforcement/#metadata-tiers) (minimal → standard → extended → temporal)
- [ ] **Migration strategy matrix** — given evolution pattern + old/new version → recommended [SCD type](/crosswalker/concepts/ontology-evolution/#how-data-warehouses-handle-this-slowly-changing-dimensions) + handling strategy
- [ ] **As-code data format research** — define primitives for representing configs, evolution profiles, and migration rules as version-controllable files
- [ ] **[Layered architecture design](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/)** — spec (universal) → library/SDK (implementations) → integrations (Obsidian plugin, CLI, framework drivers). [Architecture vision](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/)
- [ ] **[Progressive tier architecture](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/)** *(pillar)* — Tier 1 (files only + validation) → Tier 2 (files + sidecar SQLite via sql.js WASM) → Tier 3 (server-based: PocketBase, Postgres). **Files always source of truth. Each tier shares the same schemas.** This is the committed answer to Tier-1 file-canonical ceiling limits (Obsidian query-layer performance, graph-view caps, V8 heap) — we accept the ceiling at Tier 1 and scale through tiers when needed, rather than abandoning files. [Pluggable layers](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/) · [Ceiling documented in 04-10 synthesis](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#files-canonical-has-a-real-ceiling)
- [ ] **[Distribution architecture](/crosswalker/agent-context/zz-log/2026-04-03-distribution-architecture-research/)** — pnpm monorepo: `@crosswalker/core` (pure logic) + plugin (Obsidian wrapper) + CLI (Node.js `fs` wrapper). VaultAdapter interface bridges platforms. [Research](/crosswalker/agent-context/zz-log/2026-04-03-distribution-architecture-research/)
- [ ] **Graph scope decision** — directed graphs now (covers 95% of use cases). Research DAGs and hypergraph support for future extension. [Pluggable layers: Thread 2](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-2-beyond-hierarchical-taxonomies)
- [ ] **Obsidian Bases direction research** — Dataview is deprecated. Understand Bases API for building the viewing/querying layer on top of native Obsidian capabilities instead of custom UI. Datacore is a backup option to investigate if Bases turns out to be insufficient. Reconciling the "3–5K note" ceiling the research literature cites against Dataview with what Bases can actually handle is part of this item. [Context in 04-10 synthesis](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#files-canonical-has-a-real-ceiling)
- [ ] **Monorepo restructure** — extract ~50-60% of current codebase into `packages/core/`, keep plugin UI in `packages/plugin/`, create `packages/cli/`. After spec work, not before. [Sequence rationale](/crosswalker/agent-context/zz-log/2026-04-03-distribution-architecture-research/#decision-when-to-restructure)
- [ ] **[Transform engine](/crosswalker/design/transformation/)** — custom build (14 engines evaluated, none fit). 24 transform types, ChunkyCSV-style operations, optional [Arquero](https://github.com/uwdata/arquero)/[JSONata](https://jsonata.org/) escape hatches. [Ontology lifecycle: Import](/crosswalker/concepts/ontology-lifecycle/#import) · [Research](/crosswalker/agent-context/zz-log/2026-04-03-transform-engine-research/)

**Infrastructure:**

- [ ] **PII scanning in CI/CD** — automated detection of personal paths, secrets, API keys before push
- [ ] Bug fixes and polish on current MVP
- [ ] Expand E2E smoke tests minimally (don't test wizard UI — [it will change](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#6-e2e-testing-after-architecture))
- [ ] **Fluid layout scaling** — replace rigid Nova breakpoints (768/1152px) with `clamp()` CSS so spacing adapts to browser zoom. Current fix: extended breakpoint to 1152px. Full fix: fluid custom properties for `--sl-nav-height`, `--sl-content-pad-x`, typography scale. ~3-5 hours.

## Evolution — "The meta-system"

Implement the [ontology evolution](/crosswalker/concepts/ontology-evolution/) classification system designed in Foundation. This is the novel contribution — [no existing tool solves this](/crosswalker/agent-context/zz-log/2026-04-03-deep-research-synthesis/#research-1-ontology-lifecycle-management--does-anything-solve-this).

- [ ] **[Entity-aligned migration UX](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/)** — design the experience for entities (agents or people) [going through and fixing things](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/). Guided form → migration plan YAML → CLI. Progressive depth. [User-first maintenance](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/)
- [ ] **[Version registry standard](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-1-detection-is-pluggable-not-primitive)** — define the interface for declaring "framework version X exists." [Community builds implementations](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-1-detection-is-pluggable-not-primitive) (GitHub repo, API, scrapers). Detection is pluggable, not primitive.
- [ ] **[Per-framework decisioning format](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-4-decisioning-as-per-framework-taxonomy-taxonomy-over-taxonomies)** — the [taxonomy-over-taxonomies](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-4-decisioning-as-per-framework-taxonomy-taxonomy-over-taxonomies) config. Per-ontology handling strategy overrides layered on default primitives.
- [ ] **Progressive classification UX** — [community pre-classified → guided wizard → auto-detect](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#1-meta-layer-ux-progressive-classification)
- [ ] **Evolution profile registry** — YAML/JSON per framework, community-contributed
- [ ] **Migration strategy engine** — old version + new version + user evidence → suggested handling + diff preview. Built on the [structural diff engine](/crosswalker/agent-context/zz-log/2026-04-09-atomic-operations-research/).
- [ ] **Stale [crosswalk](/crosswalker/agent-context/framework-crosswalks/) detection** — surface when imported frameworks have newer versions available
- [ ] **Community marketplace research** — [Obsidian ecosystem constraints](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#8-community-marketplace-research-item), distribution model options

## Community — "Share and scale"

Build the ecosystem around the spec.

- [ ] **Community config registry** — start with GitHub repo, evaluate built-in registry later
- [ ] **[OSCAL](/crosswalker/concepts/framework-standards/) export** — export to [Open Security Controls Assessment Language](https://pages.nist.gov/OSCAL/) for GRC tool integration
- [ ] **Compliance dashboards** — [Bases views](/crosswalker/concepts/metadata-ecosystem/) for gap analysis, coverage tracking
- [ ] **Report generation** — exportable compliance reports for auditors
- [ ] **[Custom migration transforms](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-5-custom-transfer-logic)** — user-defined transformation logic for version migrations. From inline expressions to named transforms to [custom scripts](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-5-custom-transfer-logic). Progressive depth applied to migration.
- [ ] **AI-assisted transforms** — LLM-based property mapping using templates (like [Obsidian web clipper](https://obsidian.md) AI templates but for ontology node properties). Map source schema to target schema via AI suggestion.
- [ ] **Extended graph support** — research and implement DAG and [hypergraph](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-2-beyond-hierarchical-taxonomies) support beyond basic directed graphs
- [ ] **Community plugin submission** — publish to Obsidian plugin registry
- [ ] **Spec publication** — publish [EvolutionPattern taxonomy](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) and [structural diff engine](/crosswalker/agent-context/zz-log/2026-04-09-atomic-operations-research/) as standalone specifications
