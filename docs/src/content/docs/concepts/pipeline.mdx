## The funnel

Everything Crosswalker does flows through six stages. This is the core architectural artifact — every feature, every config option, every research question maps to one of these stages.

```mermaid
graph LR
    A[Ingest] --> B[Parse]
    B --> C[Transform]
    C --> D[Generate]
    D --> E[Link]
    E --> F[Evolve]
```

| Stage | What happens | Current state |
|-------|-------------|---------------|
| [Ingest](#ingest) | Acquire data from external sources | CSV file picker in wizard |
| [Parse](#parse) | Read into structured ParsedData | CSV complete, XLSX/JSON planned |
| [Transform](#transform) | Clean, normalize, restructure | Schema defined, 0% implemented |
| [Generate](#generate) | Produce folders + notes + metadata | Complete, production-ready |
| [Link](#link) | Create crosswalk relationships | Basic WikiLinks, typed links planned |
| [Evolve](#evolve) | Handle updates, migration, staleness | Research complete, implementation planned |

---

## Ingest

**What**: Acquire structured data from the outside world into the plugin.

**Current**: File picker in the [import wizard](/crosswalker/features/import-wizard/) accepting CSV files from the local filesystem.

**Planned**:
- XLSX file selection with sheet picker
- JSON/JSONL file import
- Future: URL-based import (fetch a framework CSV from a public URL)
- Future: [OSCAL](https://pages.nist.gov/OSCAL/) catalog import (machine-readable framework format)
- Future: community config [registry](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#8-community-marketplace-research-item) that bundles framework data with import configs

**Decision point**: How much do we automate acquisition? The [progressive classification UX](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#1-meta-layer-ux-progressive-classification) starts here — known frameworks could auto-fetch their data.

## Parse

**What**: Convert raw file bytes into a structured `ParsedData` object (columns + rows + metadata).

**Current**: [PapaParse](/crosswalker/reference/framework-data-sources/) for CSV with streaming support for files >5MB. [Column analysis](/crosswalker/features/import-wizard/) detects types (hierarchy, ID, text, numeric, date, tags, URL).

**Planned**:
- XLSX parser via `xlsx` package (installed, not integrated) with multi-sheet support
- JSON/JSONL parser (native)
- Column type detection improvements based on [FrameworkConfig v2](/crosswalker/agent-context/config-schema-design/) presets

**Build vs use**: Using established libraries (PapaParse, xlsx). No reason to build custom parsers — these are solved problems.

## Transform

**What**: Clean, normalize, and restructure parsed data before generation. This is where raw spreadsheet data becomes Obsidian-ready.

**Current**: [Transform schema defined](/crosswalker/design/transformation/) with 24 types (string, array, type conversion, conditional) — but **0% implemented**. The [Python tool](/crosswalker/agent-context/prior-art/) has working implementations of key transforms ([helper functions](/crosswalker/agent-context/helper-functions/)).

**The 24 transform types**:

| Category | Types |
|----------|-------|
| String | trim, lowercase, uppercase, titlecase, replace, regex_extract, prefix, suffix, template |
| Array | split, join, unique, filter, first, last, map |
| Type | to_number, to_boolean, to_date, to_tags, to_wikilinks |
| Conditional | if_empty, if_matches, coalesce, lookup |

**Per-framework transforms** (from [framework data sources](/crosswalker/reference/framework-data-sources/)):
- [Hierarchical forward-fill](/crosswalker/agent-context/helper-functions/#hierarchical_ffilldataframe-columns) — for merged Excel cells (D3FEND, CRI)
- [Tag aggregation](/crosswalker/reference/framework-data-sources/#special-transform-tag-aggregation) — CRI diagnostic statement tags
- [ID normalization](/crosswalker/reference/framework-data-sources/#assessment-procedures) — NIST assessment IDs (AC-01_ODP[01] → AC-1)
- [Preamble extraction](/crosswalker/reference/framework-data-sources/#special-transform-preamble-extraction) — CIS control descriptions
- Multi-sheet merge — MITRE ENGAGE (Goals → Approaches → Activities)

**Build vs use decision**: [Research evaluated 14 engines](/crosswalker/agent-context/zz-log/2026-04-03-transform-engine-research/) — JSONata, Arquero, Apache Arrow, DanfoJS, dbt, and others. **Verdict: build custom.** Reasons:
- ~2KB bundle vs 50-200KB for alternatives
- Obsidian-native output (WikiLinks, YAML frontmatter, vault paths) not supported by any engine
- Hierarchical folder generation not supported
- under 25ms performance for typical datasets (1000 rows)
- Full control, no breaking library updates

**Escape hatches** (optional, future): [Arquero](https://github.com/uwdata/arquero) for cross-row aggregation on huge datasets (>100K rows), [JSONata](https://jsonata.org/) for complex JSON expression transforms. See also: [Apache Arrow JS](https://arrow.apache.org/docs/js/), [DanfoJS](https://danfo.jsdata.org/), [Polars](https://pola.rs/) — all evaluated and [documented](/crosswalker/agent-context/zz-log/2026-04-03-transform-engine-research/).

**[ChunkyCSV](https://github.com/cybersader/chunkycsv)**: The user's structured-to-tabular conversion engine — search and flatten operations that could inform or augment the custom transform engine. [Research log](/crosswalker/agent-context/zz-log/2026-04-03-chunkycsv-and-transformation-engines/).

## Generate

**What**: Produce folders, markdown notes, YAML frontmatter, and [`_crosswalker` metadata](/crosswalker/agent-context/constraint-enforcement/#metadata-tiers) from transformed data.

**Current**: [Generation engine](/crosswalker/features/generation-engine/) is **complete and production-ready** (95%). Handles:
- Folder hierarchy creation from hierarchy columns
- Note content with YAML frontmatter serialization
- `_crosswalker` metadata block (sourceId, importId, configId, importedProperties)
- WikiLink and Markdown link formatting
- Template resolution (`{{columnName}}` syntax)
- Path sanitization for filesystem safety
- File existence handling (skip/replace/error modes)
- Progress callbacks for long operations

**What changes with the [spec](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/)**:
- `_crosswalker` metadata v2 (evolution profile reference, source hash, status, migration history) — [metadata tiers](/crosswalker/agent-context/constraint-enforcement/#metadata-tiers)
- Output driven by [FrameworkConfig v2](/crosswalker/agent-context/config-schema-design/) instead of raw wizard state
- Generation as a [library function](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/#layer-2-library--sdk-implementations-of-the-patterns), callable from both plugin and CLI

## Link

**What**: Create crosswalk relationships between imported frameworks and between frameworks and user evidence.

**Current**: Basic WikiLinks generated during import. No typed links, no [edge metadata](/crosswalker/agent-context/link-metadata-system/), no cross-framework crosswalking.

**Planned**:
- [Cross-framework linking engine](/crosswalker/agent-context/framework-crosswalks/) — generate typed WikiLinks from crosswalk data using [matching modes](/crosswalker/agent-context/helper-functions/#match_valuesource-target-mode) (exact, array-contains, regex)
- [Link insertion commands](/crosswalker/agent-context/link-metadata-system/) — "Insert framework link" with search modal
- [Typed link syntax](/crosswalker/agent-context/link-metadata-syntax-spec/) — `framework_here.implements:: [[AC-2]] {"sufficient": true}`
- [Evidence mapping](/crosswalker/getting-started/grc-teams/) — linking user evidence notes to framework controls with structured metadata

**Ecosystem constraint**: [Obsidian Bases](/crosswalker/concepts/metadata-ecosystem/) can query frontmatter but **cannot traverse typed links**. Edge metadata requires [DataviewJS or Datacore](/crosswalker/concepts/metadata-ecosystem/#comparison-matrix). This is a known trade-off — [documented decision](/crosswalker/agent-context/link-metadata-system/#ecosystem-compatibility).

## Evolve

**What**: Handle framework updates, version migration, [stale crosswalk detection](/crosswalker/agent-context/framework-crosswalks/#crosswalk-staleness), and long-term [data model resilience](/crosswalker/agent-context/data-model-resilience/).

**Current**: Research complete — the [ontology evolution](/crosswalker/concepts/ontology-evolution/) problem is well-understood. [No existing tool solves the meta-problem](/crosswalker/agent-context/zz-log/2026-04-03-deep-research-synthesis/#research-1-ontology-lifecycle-management--does-anything-solve-this).

**Planned (the novel contribution)**:
- [EvolutionPattern taxonomy](/crosswalker/reference/roadmap/#foundation--get-the-architecture-right) — classify how each ontology evolves (release cadence, breaking changes, ID stability, changelog format)
- [Migration strategy engine](/crosswalker/concepts/ontology-evolution/#how-data-warehouses-handle-this-slowly-changing-dimensions) — given old + new version → recommended SCD type
- [Progressive classification UX](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#1-meta-layer-ux-progressive-classification) — community pre-classified → guided wizard → auto-detect
- [Constraint enforcement](/crosswalker/agent-context/constraint-enforcement/) — lazy detection of orphaned notes, broken links, stale metadata
- Evolution profile registry — community-contributed YAML/JSON profiles per framework

**Why this matters**: Every import is a snapshot that becomes stale. The Evolve stage is what makes Crosswalker a [meta-system for ontology lifecycle management](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#core-identity-decision), not just a one-time importer.

## The pipeline as architecture

The pipeline maps directly to the [layered architecture](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/):

| Pipeline stage | Architecture layer | Who owns it |
|---------------|-------------------|-------------|
| Ingest | Integration (plugin/CLI) | Platform wrapper |
| Parse | Library (SDK) | Core library |
| Transform | Library (SDK) | Core library + transform providers |
| Generate | Library (SDK) | Core library |
| Link | Library (SDK) + Integration | Core library + Obsidian WikiLink API |
| Evolve | Spec + Library | Spec defines patterns, library implements |

The [config-as-code](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#2-config-as-code-is-the-source-of-truth) format ([FrameworkConfig v2](/crosswalker/agent-context/config-schema-design/)) configures every stage. Community [framework drivers](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/#layer-3-integrations-framework-specific-platform-specific) are JSON configs that tell the pipeline how to handle a specific framework across all six stages.
