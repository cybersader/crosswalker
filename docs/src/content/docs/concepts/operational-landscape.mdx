The operational landscape shows **who does what** across the ontology ecosystem. It combines three dimensions:
- **[Institutional actors](/crosswalker/concepts/institutional-landscape/)** — who creates, maps, regulates, consumes
- **Ontology components** — frameworks themselves vs crosswalks vs evidence vs interchange formats
- **Resource intensity** — one-time investment vs periodic updates vs continuous operational work

This is the combined view that determines what to build into software, what to crowdsource, and what requires ongoing resources (money, people, tooling, expertise).

**Reading the matrix**: Each cell shows what that actor does for that component, with links to relevant KB pages. The left border color indicates resource intensity — <span style="color:#98c379">green</span> (one-time), <span style="color:#e5c07b">yellow</span> (periodic), <span style="color:#e06c75">red</span> (continuous), <span style="color:#5c636e">gray</span> (not applicable).

Key insights from the matrix:
- **[Crosswalk maintenance](/crosswalker/concepts/ontology-evolution/#layer-2-crosswalks-break)** is red for mapping orgs, GRC vendors, AND your org — it's the most resource-intensive ongoing work across the ecosystem
- **[Evidence mapping](/crosswalker/getting-started/grc-teams/)** is your org's continuous burden — no one else can do it for you
- **[Standard bodies](/crosswalker/concepts/institutional-landscape/)** do the one-time creation work but rarely maintain crosswalks to OTHER frameworks
- **Regulators** mandate but don't create or maintain — they add requirements without resources

## The five tiers of work

### Tier 1: First-principles software (build once)

Work rooted in the fundamental nature of the problem domain. Once solved correctly, this rarely changes because the underlying principles don't change.

| Work item | Why it's resilient | Related |
|-----------|-------------------|---------|
| [EvolutionPattern taxonomy](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) | Ontologies have always evolved these ways — the patterns are structural | [Ontology evolution](/crosswalker/concepts/ontology-evolution/) |
| [Core parser/transform/generate library](/crosswalker/concepts/ontology-lifecycle/#import) | CSV/XLSX/JSON parsing is a solved problem; generation is deterministic | [Layered architecture](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/) |
| [VaultAdapter interface](/crosswalker/agent-context/zz-log/2026-04-03-distribution-architecture-research/) | The abstraction between vault operations and platform is stable | [Why files](/crosswalker/agent-context/zz-log/2026-04-03-why-obsidian-why-files/) |
| [SCD type strategies](/crosswalker/concepts/ontology-evolution/#how-to-handle-it-slowly-changing-dimensions) | Data warehouse patterns from the 1990s — proven, unchanged | [Data model resilience](/crosswalker/agent-context/data-model-resilience/) |
| [Config-as-code schema](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#2-config-as-code-is-the-source-of-truth) | JSON/YAML schema design is durable; migrations handle evolution | [Config schema](/crosswalker/agent-context/config-schema-design/) |

**Business analogy**: This is R&D. High upfront cost, near-zero marginal cost afterward.

### Tier 2: Refined software (evolves slowly)

Built on first principles but adapts as the ecosystem shifts. Changes on the order of years, not months.

| Work item | What causes change | Related |
|-----------|-------------------|---------|
| [Transform engine](/crosswalker/agent-context/zz-log/2026-04-03-transform-engine-research/) (24 types) | New data patterns emerge; new framework formats appear | [Ontology lifecycle: Import](/crosswalker/concepts/ontology-lifecycle/#import) |
| [Migration strategy engine](/crosswalker/concepts/ontology-evolution/) | New SCD type combinations discovered through practice | [Constraint enforcement](/crosswalker/agent-context/constraint-enforcement/) |
| Plugin UI (wizard, settings) | Obsidian API changes; UX improvements from user feedback | [Roadmap: Formats](/crosswalker/reference/roadmap/#formats--import-anything-transform-it-properly) |
| [Typed link syntax](/crosswalker/agent-context/link-metadata-syntax-spec/) | [Metadata ecosystem](/crosswalker/concepts/metadata-ecosystem/) evolves (Bases, Datacore) | [Link metadata](/crosswalker/agent-context/link-metadata-system/) |

**Business analogy**: This is product engineering. Continuous improvement, but the foundation doesn't change.

### Tier 3: Community-maintained (shared work)

Work that's per-framework but shareable. Once someone configures a framework, everyone benefits. This is where community leverage matters most.

| Work item | Who does it | Related |
|-----------|------------|---------|
| [Evolution profiles](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) (per-framework YAML) | Community contributes; Crosswalker validates | [EvolutionPattern spec](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) |
| [FrameworkConfig presets](/crosswalker/agent-context/config-schema-design/) (import configs) | First user configures; shares via [registry](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#8-community-marketplace-research-item) | [Config schema](/crosswalker/agent-context/config-schema-design/) |
| [Crosswalk mapping data](/crosswalker/agent-context/framework-crosswalks/) | Often sourced from [OLIR, SCF, CTID](/crosswalker/concepts/institutional-landscape/#the-entities); community refines | [Framework crosswalks](/crosswalker/agent-context/framework-crosswalks/) |

**Business analogy**: This is the marketplace/ecosystem. The more contributors, the more valuable each contribution becomes. Like npm packages or Terraform providers.

**Key insight**: This tier is where Crosswalker's [community config registry](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#8-community-marketplace-research-item) would create the most value — the hard per-framework work done once, shared infinitely.

### Tier 4: Operational maintenance (ongoing)

Work that recurs because the external world changes. Cannot be eliminated — only automated or made more efficient.

| Work item | Why it recurs | Related |
|-----------|-------------|---------|
| Framework version tracking | [Framework owners](/crosswalker/concepts/institutional-landscape/#the-entities) publish updates on their own schedules | [Framework versioning](/crosswalker/concepts/framework-versioning/) |
| Crosswalk refresh | When either framework in a crosswalk updates, the [mapping may go stale](/crosswalker/concepts/ontology-evolution/#layer-2-crosswalks-break) | [Crosswalk staleness](/crosswalker/agent-context/framework-crosswalks/#crosswalk-staleness) |
| Stale detection notifications | Users need to know when their imported snapshot is outdated | [Constraint enforcement](/crosswalker/agent-context/constraint-enforcement/) |
| Community config validation | Submitted configs need review, testing, version bumping | [Progressive classification](/crosswalker/agent-context/zz-log/2026-04-03-vision-alignment-decisions/#1-meta-layer-ux-progressive-classification) |

**Business analogy**: This is DevOps / SRE. If you built a business, this is where subscription revenue would fund ongoing labor. The [institutional timing problem](/crosswalker/concepts/ontology-evolution/#the-timing-problem) means this work literally never ends.

### Tier 5: User work (your team's effort)

Work that's specific to YOUR organization. No tool can eliminate it — only make it more efficient.

| Work item | Why only you can do it | Related |
|-----------|----------------------|---------|
| [Evidence mapping](/crosswalker/getting-started/grc-teams/) | Only your team knows which policies satisfy which controls | [GRC teams](/crosswalker/getting-started/grc-teams/) |
| Assessment documentation | Only your team knows your implementation details | [Evidence mapping](/crosswalker/getting-started/grc-teams/#map-evidence-with-structured-metadata) |
| Audit trail maintenance | Only your team can answer auditor questions | [Institutional landscape](/crosswalker/concepts/institutional-landscape/#the-relationships) |
| Custom crosswalks | Your org may have unique mapping needs between internal and external ontologies | [Schema matching](/crosswalker/concepts/schema-matching/) |

**Business analogy**: This is the customer's work. The tool makes it faster and more reliable, but it's fundamentally human judgment.

## Why this matters architecturally

The gradient directly maps to the [layered architecture](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/):

| Tier | Architecture layer | Investment type |
|------|-------------------|----------------|
| Tier 1 (first principles) | [Spec](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/#layer-1-spec-universal-first-principled) | One-time R&D |
| Tier 2 (refined software) | [Library/SDK](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/#layer-2-library--sdk-implementations-of-the-patterns) | Continuous engineering |
| Tier 3 (community) | [Framework drivers](/crosswalker/agent-context/zz-log/2026-04-03-layered-architecture-vision/#layer-3-integrations-framework-specific-platform-specific) | Ecosystem building |
| Tier 4 (operations) | Registry/infrastructure | Ongoing operations |
| Tier 5 (user work) | Plugin UX | Customer success |

**The goal of Crosswalker's architecture**: Push as much work as possible toward Tier 1 (solved once, resilient forever). Where that's impossible (Tiers 3-4), build community infrastructure so the work is shared. Only Tier 5 is irreducibly the user's burden.

---

## The sustainability question

If this were a business:

- **Tiers 1-2** are funded by initial development (open source contributions, grants, or VC)
- **Tier 3** is self-sustaining once the community is large enough (network effects)
- **Tier 4** is the ongoing cost that requires either subscription revenue, sponsorship, or dedicated volunteers
- **Tier 5** is the value proposition — you sell efficiency, not elimination, of this work

The [EvolutionPattern taxonomy](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) is a Tier 1 investment that reduces Tier 4 costs — by classifying evolution patterns upfront, stale detection and migration become more automated and less human-dependent.
