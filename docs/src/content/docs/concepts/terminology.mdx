## Core terms

| Term | Definition |
|------|-----------|
| **Ontology** | A structured representation of knowledge — categories, relationships, and rules |
| **Framework** | A specific ontology (NIST 800-53, CIS Controls, etc.) |
| **Crosswalk** | A mapping between concepts in two structured ontologies — showing how elements in one correspond to elements in another. Examples: `NIST 800-53/AC-2` ↔ `ISO 27001/A.9.2.1` (framework-to-framework), `NIST AC-2` → `SCF/canonical-AM-01` (framework-to-[synthetic spine](/crosswalker/agent-context/zz-challenges/06-synthetic-spine/)). **Crosswalks are a specific kind of edge where both ends are ontology concepts.** Distinct from **evidence links** (see below), which connect user-authored documents to ontology concepts. For the technical vocabulary and metadata envelope Crosswalker commits to for crosswalk edges, see [STRM](/crosswalker/reference/registry/strm/) + [SSSOM](/crosswalker/reference/registry/sssom/). For the architectural pattern, see the [schema matching interlingua / pivot approach](/crosswalker/concepts/schema-matching/#the-interlingua--pivot-approach). |
| **Evidence link** — also: implementation link, junction note (storage shape), typed link (evidence case), edge metadata (evidence case), link metadata, dotKey metadata (legacy) | An edge from a user-authored document to an ontology concept — e.g., `MFA-Policy.md` → `NIST 800-53/AC-2` with properties like `status: implemented`, `confidence: 0.85`, `reviewer: Alice`, `review_date: 2026-04-10`. **Not a crosswalk.** The semantic is "this document demonstrates/implements this control at such-and-such status," not "these two concepts correspond." **Foundation direction committed** (pending formal lock-in of open sub-decisions): stored as **junction notes** (one markdown file per edge) with a 13-field flat-YAML schema isomorphic to [OSCAL](/crosswalker/reference/registry/oscal/)'s `by-component` assembly. Three orthogonal status dimensions: `status` (5 OSCAL values) + `freshness` (4 computed values) + `evidence_type` (8 values). Git history is the audit trail. See the [**2026-04-10 evidence-link edge model synthesis log**](/crosswalker/agent-context/zz-log/2026-04-10-evidence-link-edge-model-synthesis/) for the full decision, [Challenge 07](/crosswalker/agent-context/zz-challenges/07-link-metadata-edge-model/) for the research brief, and the [roadmap evidence-link edge model item](/crosswalker/reference/roadmap/) for the commitment tracking + open sub-decisions. The original [link metadata system](/crosswalker/agent-context/link-metadata-system/) inline-Dataview syntax is **superseded** by the junction note approach. |
| **Hierarchy column** | A data column that defines folder nesting levels |
| **Frontmatter** | YAML properties at the top of a markdown note |
| **Typed link** | A link with metadata describing the relationship type. Umbrella term that covers both crosswalks and evidence links as specific sub-kinds. |
| **Fingerprint** | A hash of a data file's structure used for config matching |

| Term | Definition |
|------|-----------|

## Ontology evolution terms

These come up throughout the [ontology evolution](/crosswalker/concepts/ontology-evolution/) and [framework versioning](/crosswalker/concepts/framework-versioning/) pages.

| Term | Definition |
|------|-----------|
| **Ontology evolution** | How a structured knowledge system changes over time — adding, removing, or restructuring concepts. Distinct from ontology *versioning* (tracking which version you have). [More](/crosswalker/concepts/ontology-evolution/) |
| **Schema evolution** | Changes to the structure of data (adding fields, renaming columns) as opposed to changes to the data itself |
| **Deprecation chain** | A sequence of `revoked-by` relationships linking old concepts to their replacements. MITRE ATT&CK uses STIX 2.1 for this — when technique T1234 is deprecated, the chain says "replaced by T5678." [More](/crosswalker/concepts/framework-versioning/#mitre-attck--the-gold-standard) |
| **SemVer** | Semantic Versioning (MAJOR.MINOR.PATCH). OSCAL uses this with a guarantee: content created under a MAJOR version remains valid in later releases within that major. [More](/crosswalker/concepts/ontology-evolution/#schemaver-an-alternative-to-semver-for-data) |
| **SchemaVer** | MODEL-REVISION-ADDITION versioning for data (from Snowplow). MODEL = can old consumers read new data? REVISION = can new consumers read old data? ADDITION = non-breaking new fields. [More](/crosswalker/concepts/ontology-evolution/#schemaver-an-alternative-to-semver-for-data) |
| **SCD (Slowly Changing Dimensions)** | Data warehouse pattern for handling records that change over time. Type 1 = overwrite, Type 2 = keep history (version-tagged folders), Type 3 = add alias column, Type 6 = hybrid. [More](/crosswalker/concepts/ontology-evolution/#how-data-warehouses-handle-this-slowly-changing-dimensions) |
| **[SKOS](/crosswalker/reference/registry/skos/)** | Simple Knowledge Organization System (W3C). Natural fit for frameworks: ConceptScheme = framework, Concept = control, broader/narrower = hierarchy, exactMatch = crosswalk. The 5-predicate baseline vocabulary (exactMatch/closeMatch/broadMatch/narrowMatch/relatedMatch) — found *insufficient* for compliance crosswalks in the [04-10 synthesis](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#edge-semantics-should-be-set-theoretic-not-skos-ish) (no confidence, provenance, or negation). [Registry entry](/crosswalker/reference/registry/skos/) · [More in schema matching](/crosswalker/concepts/ontology-evolution/#skos-the-natural-formal-model) |
| **[SSSOM](/crosswalker/reference/registry/sssom/)** | Simple Standard for Sharing Ontological Mappings (Matentzoglu et al. 2022). The **row-schema envelope** for crosswalk edges: mandatory `mapping_justification` + optional `confidence`, `author_id`, `mapping_date`, `mapping_tool`, `predicate_modifier` for negation. Predicate-agnostic (you choose the vocabulary for `predicate_id` — see STRM). Crosswalker's candidate edge metadata model. [Registry entry](/crosswalker/reference/registry/sssom/) · [Schema matching](/crosswalker/concepts/schema-matching/#sssom-a-standard-for-sharing-mappings) |
| **[STRM](/crosswalker/reference/registry/strm/)** — Set Theory Relationship Mapping | The 5-relationship edge vocabulary — **equivalent-to, subset-of, superset-of, intersects-with, no-relationship** — that [NIST IR 8477](https://csrc.nist.gov/pubs/ir/8477/final) (Feb 2024), [SCF](/crosswalker/reference/registry/scf/)'s STRM bundle, and OSCAL's Control Mapping Model independently converged on. Crosswalker's candidate `predicate_id` vocabulary (paired with SSSOM as the envelope). Fills the gaps SKOS leaves open. [Registry entry](/crosswalker/reference/registry/strm/) · [NIST OLIR formal relationship types](/crosswalker/concepts/schema-matching/#nist-olirs-formal-relationship-types) · [Foundation decision](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#edge-semantics-should-be-set-theoretic-not-skos-ish) |
| **OSCAL** | Open Security Controls Assessment Language (NIST). Machine-readable format for security controls with built-in versioning. Also adopts STRM relationships in its emerging Control Mapping Model. [Registry entry](/crosswalker/reference/registry/oscal/) · [More](https://pages.nist.gov/OSCAL/) |
| **[OLIR](/crosswalker/reference/registry/olir/)** | Online Informative References (NIST). Submission-based crosswalk registry with formal relationship types (Subset, Intersects, Equal, Superset, Not Related). Uses the same STRM vocabulary NIST IR 8477 formalized. [Registry entry](/crosswalker/reference/registry/olir/) · [Formal relationship types](/crosswalker/concepts/schema-matching/#nist-olirs-formal-relationship-types) |
| **Interlingua / pivot** — also: pivot ontology, synthetic spine, hub-and-spoke mapping, meta-framework | A "universal framework" that others map through — like a pivot language in translation. [SCF](/crosswalker/reference/registry/scf/)'s STRM is this for compliance: map to SCF once, get crosswalks to 175+ frameworks. These five terms all describe the same architectural pattern from different disciplines (translation theory, information integration, GRC practice). Open Foundation question — see [Challenge 06](/crosswalker/agent-context/zz-challenges/06-synthetic-spine/) and the [04-10 synthesis spine section](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#the-synthetic-spine-is-the-missing-architectural-insight). [Schema matching: interlingua / pivot approach](/crosswalker/concepts/schema-matching/#the-interlingua--pivot-approach) |

## Architecture & evolution primitives

These terms emerged from the [Foundation phase](/crosswalker/reference/roadmap/) research. Each has aliases that appear across the knowledge base.

| Term | Aliases | Definition |
|------|---------|-----------|
| **Ontology diff primitives** | Graph change atoms, structural change primitives, structural diff engine | The [9 provably complete atomic operations](/crosswalker/agent-context/zz-log/2026-04-09-atomic-operations-research/) for describing what changed between two versions of a labeled directed graph. The mathematical foundation — "what changed?" Based on graph edit distance (Bunke 1983) and algebraic graph transformation (Ehrig 2006). |
| **EvolutionPattern** | Framework handling profile, stewardship profile, evolution taxonomy | Per-framework classification of how an ontology evolves. Sets defaults for [handling strategies](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/#step-3-decision). **Open question:** may be replaced or complemented by [transformation recipes](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/). [Original draft](/crosswalker/agent-context/zz-log/2026-04-03-evolution-pattern-taxonomy-draft/) |
| **Transformation recipe** | Version transition map, migration spec | The actual record of what changed between specific versions of a framework and how to handle each change. E.g., "NIST r5→r6: AC-19(2) merged into AC-19, AC-24 added." May replace or complement EvolutionPattern. |
| **Handling strategy** | — | What to do when a specific [change type](/crosswalker/agent-context/zz-log/2026-04-08-ontology-evolution-first-principles/) is detected: overwrite, archive, alias, merge, split, skip, or flag for review. |
| **Decisioning** | Strategy selection | The process of selecting handling strategies per change type. Can be default (from primitives), per-framework (from EvolutionPattern), or per-user/org (custom overrides). [Layered model](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-4-decisioning-as-per-framework-taxonomy-taxonomy-over-taxonomies) |
| **Pluggable layer** | Extension point, opinionated layer | Any system behavior that is opinionated and swappable rather than first-principles. [Detection](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-1-detection-is-pluggable-not-primitive), decisioning, and [custom transforms](/crosswalker/agent-context/zz-log/2026-04-09-primitives-depth-and-pluggable-layers/#thread-5-custom-transfer-logic) are all pluggable. Keeps the system "[tight to first principles while allowing extension of opinionation](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/#the-first-principles-constraint)." |
| **Progressive depth** | Progressive complexity | Design principle: start with guided UI, graduate to config-as-code, add CLI. Each interface wraps the same [core primitives](/crosswalker/agent-context/zz-log/2026-04-09-atomic-operations-research/). Applies to both [import](/crosswalker/features/import-wizard/) and [migration](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/) workflows. |
| **Entity** | User, agent | Any actor utilizing the system — [agents or people](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/#the-entities). The system must be [efficacious for all entity types](/crosswalker/agent-context/zz-log/2026-04-09-user-first-ontology-maintenance/). |

## Data model terms

These appear in the [file-based graph database](/crosswalker/concepts/file-based-graph-database/) and [consistency models](/crosswalker/concepts/consistency-models/) pages.

| Term | Definition |
|------|-----------|
| **Property graph** | A graph where both nodes and edges carry key-value properties. Obsidian vaults are property graphs: notes = nodes, WikiLinks = edges, frontmatter = properties. [More](/crosswalker/concepts/file-based-graph-database/#property-graph-model) |
| **Edge metadata** — also: link metadata, typed link metadata, inline field metadata, dotKey metadata | Structured data attached to a relationship (link) between two notes, not to the notes themselves. The umbrella data-model concept behind both [crosswalks](#core-terms) and [evidence links](#core-terms). Original design: [link metadata system](/crosswalker/agent-context/link-metadata-system/) (superseded — see its caution callout). **Both edge categories now have committed directions**: crosswalks use [STRM + SSSOM](/crosswalker/reference/roadmap/) (ontology↔ontology, set-theory predicates + provenance envelope), evidence links use [junction notes with a 13-field flat-YAML schema](/crosswalker/agent-context/zz-log/2026-04-10-evidence-link-edge-model-synthesis/) (document→control, OSCAL `by-component` isomorphic). |
| **Denormalization** | Deliberately duplicating data to speed up reads at the cost of write complexity. When Crosswalker stores both forward links and frontmatter arrays, that's denormalization. [More](/crosswalker/concepts/file-based-graph-database/#the-denormalization-tax) |
| **Materialized view** | Pre-computed query results stored for fast access. Dataview/Bases queries over vault files are materialized views — they compute on each query, never getting stale. |
| **Referential integrity** | A guarantee that all references (links) point to existing targets. Obsidian vaults have NO referential integrity — you can delete a note and leave broken WikiLinks everywhere. [More](/crosswalker/agent-context/constraint-enforcement/#the-referential-integrity-gap) |
| **ACID** | Atomicity, Consistency, Isolation, Durability — database transaction guarantees. File-based systems only provide Durability. [More](/crosswalker/concepts/consistency-models/#acid-analysis-for-file-based-systems) |
| **CAP theorem** | A distributed system can provide at most two of: Consistency, Availability, Partition tolerance. Obsidian vaults are AP systems (available + partition-tolerant, not strongly consistent). [More](/crosswalker/concepts/consistency-models/#cap-theorem-positioning) |
| **Eventual consistency** | Given enough time, all copies of data converge to the same state. In Crosswalker: re-import achieves consistency, but it requires manual action. [More](/crosswalker/concepts/consistency-models/#base-model) |
| **Event sourcing** | Storing the sequence of changes (events) rather than just the current state. The `_crosswalker.history` field (future) would enable this. [More](/crosswalker/concepts/consistency-models/#event-sourcing-pattern) |

## Matching & alignment terms

From the [schema matching](/crosswalker/concepts/schema-matching/) page.

| Term | Definition |
|------|-----------|
| **Schema matching / schema alignment / ontology matching / ontology alignment / crosswalking / data integration / framework mapping** | *Not separate things — all names for the same broad concept.* Finding correspondences between elements of different structured datasets. Each name comes from a different discipline: schema matching = database community (Rahm & Bernstein 2001), ontology matching/alignment = Semantic Web (Euzenat & Shvaiko 2013), crosswalking = library science (Woodley 2008), data integration = ETL/warehousing, framework mapping = GRC practitioners. Within a single field the terms have precise distinct meanings (see [terminology distinctions table](/crosswalker/concepts/schema-matching/#terminology-distinctions)), but **Crosswalker straddles all these fields** — which name you see on any given KB page depends on which community that page is drawing from. [Aliases callout on schema-matching page](/crosswalker/concepts/schema-matching/) |
| **Schema matching** (narrow sense) | Database-community term: finding corresponding *elements* (columns, fields, keys) between schemas. Output is a set of element pairs. Crosswalker's column-type detection is a simple example. |
| **Schema mapping** (narrow sense) | The executable *transformation* that implements a match — "split Column A by comma, map each to Column B." |
| **Ontology alignment** (narrow sense) | Semantic-web term: finding corresponding *concepts* between ontologies with confidence scores. Output is an "alignment" (set of correspondences). When NIST says "AC-2" corresponds to CIS "5.2 at 0.85 confidence," that's ontology alignment. |
| **Ontology merging** | Combining two ontologies into one unified ontology. SCF combining 175+ frameworks into a single canonical set is an example. |
| **Crosswalk edge semantics stack** | Crosswalker-specific term for the layered commitment: [**STRM**](/crosswalker/reference/registry/strm/) as the required `predicate_id` vocabulary + [**SSSOM**](/crosswalker/reference/registry/sssom/) as the required metadata envelope + [**SKOS**](/crosswalker/reference/registry/skos/) rejected-as-base but kept as an export format via the YAML-LD bridge. [Foundation decision](/crosswalker/agent-context/zz-log/2026-04-10-foundation-research-synthesis/#edge-semantics-should-be-set-theoretic-not-skos-ish) · [Roadmap pillar](/crosswalker/reference/roadmap/) |
| **OAEI** | Ontology Alignment Evaluation Initiative — annual competition benchmarking alignment algorithms. Recent LLM-based systems achieve 0.83-0.95 F1 scores. [More](/crosswalker/concepts/schema-matching/) |
| **Formal Concept Analysis** | Lattice theory approach to finding structural correspondences between concept hierarchies without relying on string similarity. Candidate mechanism for [spine distillation](/crosswalker/agent-context/zz-challenges/06-synthetic-spine/). [More](/crosswalker/concepts/schema-matching/#formal-concept-analysis-fca) |

## Metadata ecosystem terms

From the [metadata ecosystem](/crosswalker/concepts/metadata-ecosystem/) page.

| Term | Definition |
|------|-----------|
| **Inline field** | Dataview syntax: `key:: value` in note body. Only queryable by Dataview (and potentially Datacore), NOT by Obsidian Bases. |
| **Properties** | Obsidian's native YAML frontmatter system (since v1.4). First-class, queryable by Bases and Dataview. |
| **Obsidian Bases** | Native database views (.base files). Tabular only — can query frontmatter but CANNOT traverse typed links or query inline fields. [More](/crosswalker/concepts/metadata-ecosystem/) |
| **Datacore** | Community successor to Dataview (in development). Different API, better performance. |
